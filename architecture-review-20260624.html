1. Extract a shared fs_transaction module

Collapse triplicated snapshot/rollback primitives behind a single deep module.

Files

apply.py sync.py workspace_activation.py backups.py skill_fs.py

Duplication Count

_PathSnapshot class3 copies
_snapshot_path3 copies
_restore_snapshot(s)3 copies
_remove_path4 copies
_unique_paths3 copies

Problem

Every write-path module (apply, sync, workspace activation) implements its own nearly identical snapshot-and-rollback mechanism. These five private helpers appear 3–4 times each, with only tiny variations. A bug fix or edge-case improvement to rollback logic must be replicated in three places, and drift between copies is a silent correctness risk during failure recovery — exactly the scenario where consistency matters most. The deletion test fails: deleting any one copy does not make complexity vanish; it forces callers to re-implement it.

Before / After

BEFORE — Shallow copies, poor locality

AFTER — Deep module, single seam

Solution

Create a new fs_transaction.py module exposing a small interface — a transaction(snapshot_paths, body) context manager plus atomic_replace_dir. All three write-path modules call into it. The existing private helpers in apply.py, sync.py, and workspace_activation.py are deleted. skill_fs.py's replace_skill_folder_atomic becomes a thin adapter or moves into the same module.

Benefits (leverage & locality)

  • ▸ Locality: rollback bugs fixed in one file, not three.
  • ▸ Leverage: 1 test surface instead of 3; new write paths get atomic rollback for free.
  • ▸ Testability: transactional FS semantics unit-tested once with temp dirs.
  • ▸ Deletion test: deleting fs_transaction would force all three callers to re-implement rollback — it earns its keep.
Strong Recommendation

2. Introduce a Host Adapter seam

Replace 26+ scattered if host == "codex" checks with a two-adapter seam.

Files

planner.py apply.py cli.py workspace_activation.py host_paths.py backups.py codex_config.py _archive.py

Host Branch Count

apply.py9 branches
cli.py6 branches
planner.py3 branches
workspace_activation.py5 branches
Other modules3 branches

Problem

Host differences (Codex config-write vs Claude archive-move) leak into every layer. The planner produces different operation lists per host, apply contains different execution phases per host, and validation contains host-specific guards. This is a real seam — two concrete adapters already exist (codex_config.py and _archive.py) — but the dispatch logic is spread across 7+ modules rather than concentrated behind an interface. Adding a third host (e.g. Cursor, Windsurf) would require touching every module.

Solution

Define a HostAdapter interface that captures host-varying operations: plan_operations(), execute_disable(), skill_root_namespace(), backup_targets(), validate_for_host(). Two adapters implement it: CodexAdapter (wrapping codex_config.py) and ClaudeAdapter (wrapping _archive.py). Planner, apply, workspace_activation take a HostAdapter and call its interface instead of branching on string comparisons.

Benefits

  • ▸ Locality: host-specific logic moves to one file per host.
  • ▸ Leverage: adding a third host is one new adapter, not seven file edits.
  • ▸ Testability: adapters tested in isolation; core planner/apply become host-agnostic.
  • ▸ Real seam: two adapters already exist — the principle says this is a real seam.

3. Consolidate validation helpers through path_safety

Five duplicated _safe_component copies; one thorough implementation already exists.

Files with duplicated validators

planner.py apply.py cli.py workspace_activation.py pack_inspect.py pointer.py

Canonical module exists

path_safety.py

Problem

path_safety.safe_component() implements the most thorough checks (Windows reserved names, colons, trailing dots/spaces, length limits). Yet five other modules each define their own weaker _safe_component, and two more define _safe_pointer_skill. The same applies to _ensure_under (3 copies), _required_path (3 copies), _operations_of_kind (3 copies), and _single_operation (2 copies). These are not deep — they are pass-throughs that exist because the canonical module is not used consistently.

Solution

Make path_safety.py the single seam for all path/component validation. Add safe_pointer_skill(), ensure_under(), require_path(), and plan helpers like operations_of_kind(). Delete all private duplicates. Callers import from the canonical module.

Benefits

  • ▸ Locality: one place to fix path-safety bugs or tighten validation.
  • ▸ Security: no weak-validator bypass — all names go through thorough checks.
  • ▸ Test surface: one test module instead of six; reduces ~120 lines of duplication.

Problem

CLI handlers should be thin: parse args → call library → print output. But cli.py contains significant domain logic — selection validation, fingerprint computation, plan context extraction, path redaction, backup metadata annotation. These functions are untestable through the CLI interface (they require argparse setup and stdout capture) even though they contain pure logic that could be unit-tested directly. The CLI module is 939 lines when it should be ~400.

Solution

Move pure domain logic out of cli.py into the modules that own those concepts: selection validation → recommendation_store.py; manifest fingerprint → manifest.py; plan context extraction → planner.py; path redaction → a small new redaction.py. CLI handlers become thin wrappers that call the library and format output.

Benefits

  • ▸ Testability: domain logic becomes directly unit-testable without CLI harness.
  • ▸ Locality: recommendation validation lives near the recommendation store, not the CLI parser.
  • ▸ Leverage: other entry points (future Python API or MCP server) can call the same logic.

5. Split backups.py (849 LOC) into record management vs restore execution

The largest module mixes three concerns: backup creation, metadata/record-keeping, and restore execution.

Concerns mixed in backups.py

BACKUP CREATION
create_backup, snapshot logic
RECORD / METADATA
list_backups, prune_backups, metadata read/write
RESTORE EXECUTION
restore_backup, rollback, Claude archive restore

Problem

At 849 lines, backups.py is the largest module. It mixes three concerns that change for different reasons: listing/pruning backups (read-only), creating backups (write on apply), and restoring backups (write on restore). It also has its own snapshot helper and a lazy import of _archive (suggesting circular dependency pressure). Claude-specific archive-restore entries are embedded here rather than behind a host adapter.

Speculative because it depends on candidate #2 (host adapter) being done first.

Solution

After the host adapter seam exists, split backups.py into: backup_records.py (list, prune, metadata I/O), backup_create.py (creation, leveraging fs_transaction), and restore.py (restore execution, delegating to the adapter).

Benefits

  • ▸ Locality: backup listing tests don't import restore execution machinery.
  • ▸ Depth: each module has a smaller interface and clearer responsibility.
  • ▸ Circular deps: splitting may resolve the lazy-import pattern.

The fs_transaction extract is the ideal starting point because:

  1. No interface design risk — the existing three copies reveal the exact interface needed. The shape of the seam is already proven by usage.
  2. No semantic changes — this is a pure refactoring. The new module is a mechanical extract with existing tests as the safety net.
  3. Unblocks candidate #2 — once atomic FS operations are behind a single seam, the host adapter can compose with it cleanly.
  4. Deletion test passes loudly — removing fs_transaction would force every write-path module to re-implement snapshot/rollback. It earns its keep immediately.
  5. Improves AI-navigability — currently an agent exploring rollback behavior must read 3 files; after the extract, one module contains the entire story.

Recommended sequence: fs_transaction → host adapter → validation consolidation → CLI slimming → backups split. Each step is independently mergeable and leaves the codebase in a better state.
